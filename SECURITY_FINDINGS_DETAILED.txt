ISSUE #4: SQL INJECTION VIA UNVALIDATED ILIKE PATTERN
================================================================================
Severity: HIGH
Type: SQL Injection (Potential)
File: /Users/usangaraju/Downloads/smartfarm/backend/routers/activity_log.py
Line: 40

CURRENT CODE:
    if action:
        q = q.filter(ActivityLog.action.ilike(f"%{action}%"))

ANALYSIS:
- SQLAlchemy's ilike() DOES parameterize, so injection is NOT currently possible
- However, the pattern is a code smell indicating potential future vulnerabilities
- If code is refactored to use string concatenation, it WILL be vulnerable

RISK:
- Medium now (SQLAlchemy protects), but High if code refactoring occurs
- Developers may copy this pattern thinking it's safe everywhere
- Database query logs show actual values in SQL (information disclosure)

CURRENT STATUS: SAFE (due to SQLAlchemy)

RECOMMENDED FIX (Best Practice):
    # Option 1: Validate against enum
    ALLOWED_ACTIONS = {"LOGIN", "LOGIN_FAILED", "CREATE_USER", "UPDATE_USER", ...}

    if action:
        if action.upper() not in ALLOWED_ACTIONS:
            raise HTTPException(400, f"Invalid action: {action}")
        q = q.filter(ActivityLog.action == action.upper())

    # Option 2: If free-text search is required, still use LIKE with parameterization
    if action:
        q = q.filter(ActivityLog.action.ilike(f"%{action.strip()}%"))

REASONING:
- Enum validation prevents injection AND improves API clarity
- Allows clients to discover valid values
- Free-text LIKE queries are slow anyway; enum is better
